# Микро-навык: решение на базе 514 (ОНМС + опц. портфель)

**Курс / канон:** [[514-CM-MIPT]] — материалы 12-CM-STAI.

**Опорный канон** — этот `SKILL.md`: ОНМС-таблица, формула A/B/C/D и каноническая секция `### Дерево решений` заданы ниже и достаточны для применения навыка. В полном vault Master-Strategy2 / STAI есть глубже проработанные `514-Skill-Реализация.md` (ОНМС) и `514-Skill-Решение.md` (Effort/Impact, рычаги Медоуз) — они опциональны.

Полный pipeline проблемы — [.cursor/skills/stai-problem-solving/SKILL.md](.cursor/skills/stai-problem-solving/SKILL.md) и [.claude/commands/solve-problem.md](.claude/commands/solve-problem.md).

## Light-режим в 504-demo (без карточек проектов)

В vault 504-demo **нет карточек проектов** и нет секции «Ключевые решения» ПЗ. Протокол MD-6 (`### Дерево решений` в карточке проекта) **не применяется** буквально. Вместо него:

1. **Решение по задаче дня** → блок в daily-заметке `2026/01-Daily/YYYY-MM-DD.md`:
   ```
   ### Решение HH:mm
   **A, а не B, потому что C, чтобы D.**
   
   | Поле | Значение |
   |---|---|
   | Обоснованность | … |
   | Необратимость | … |
   | Масштаб | … |
   | Субъектность | … |
   ```

2. **Решение с накоплением (журнал)** → отдельный файл `My-Notes/decisions-log.md`. При каждом решении дописывать запись с датой-заголовком и той же ОНМС-таблицей. Удобно для ретроспективного обзора паттернов решений.

3. **Решение, привязанное к модулю курса** → в `My-Notes/M{N}-HW-Protocol.md` в секции «Решения» или рядом с обсуждением, куда оно логически относится.

Протокол MD-0…MD-5 (горизонт, вопрос, варианты, ОНМС, формула A/B/C/D) применяется как есть. MD-6 адаптируется под путь выше. MD-7 (стратегия / задача) — по запросу пользователя.

## Когда применять

По **явному запросу** или когда пользователь застрял между 2+ вариантами и нужна **структура выбора без полного STAI**.

**Триггеры (RU):** принять решение, зафиксировать решение, выбрать вариант, оценить ставку, дерево решений, ОНМС, микро-решение 514.

**Триггеры (EN):** decide, choose between, fix the decision, decision tree, bet evaluation.

**Не подменять этим навыком:**

- Полный цикл «проблема → диагностика → портфель» — **stai-problem-solving**.
- Обзор стратегического документа блоками — Strategy Review ([.claude/commands/strategy-review.md](.claude/commands/strategy-review.md)) и [.cursor/rules/strategy-framework.md](.cursor/rules/strategy-framework.md).
- Создание/рефакторинг Objectives и карточки проекта целиком — [project-management](.cursor/skills/project-management/SKILL.md) / [refactoring-objective](.cursor/skills/refactoring-objective/SKILL.md) (здесь только **вставка текста решения** по запросу).

## Алгоритм

### MD-0. Горизонт

Уточнить (или вывести из контекста и подтвердить одним вопросом):

- **Стратегия** — ставка уровня направления / портфеля / рынка.
- **Проект** — выбор в рамках ProjectId (есть путь к карточке проекта или ПЗ).
- **Задача** — Objective / MIO / тактический шаг.

### MD-1. Вопрос решения

Коротко зафиксировать: *что именно решаем одним выбором?*

### MD-2. Варианты

Минимум **два** варианта, включая явное **«не делать»** / «отложить», если уместно.

### MD-3. Опционально: слой Skill 3

Если вариантов несколько и нужна приоритизация — кратко разметить каждый:

- **Уровень рычага** (1–5 Медоуз) или `n/a` для бытовых решений.
- **Effort** / **Impact** (низкий / средний / высокий).

**Карта стейкхолдеров** из Skill 3 — только если пользователь просит или явный конфликт интересов.

### MD-4. ОНМС по выбранному варианту

Заполнить таблицу (формулировки вопросов — как в каноне 514):

| Поле | Вопрос |
| ---- | ------ |
| **Обоснованность** | На чём основана ставка? Какие данные и модели поддерживают? |
| **Необратимость** | Насколько легко откатить? (высокая необратимость = больше проверки до запуска) |
| **Масштаб** | Какой потенциальный эффект — на продукт / на рынок / на экосистему? (на уровне проекта/задачи — сузить: на цель проекта / на результат недели) |
| **Субъектность** | Кто реализует? Личный / командный / организационный уровень? |

### MD-5. Формула фиксации A / B / C / D (обязательно)

После согласования с пользователем **всегда** вывести одну строку:

**Шаблон:** «Выбираем **A**, а не **B**, потому что **C**, чтобы **D**.»

| Слот | Смысл | Связь с ОНМС |
| ---- | ----- | ------------- |
| **A** | Принятый вариант | Что делаем |
| **B** | Отвергнутый (или «не делать X») | Явный trade-off, согласуется с идеей стратегии как выборов и «NO» ([strategy-framework](.cursor/rules/strategy-framework.md)) |
| **C** | Почему так | Прежде всего обоснованность; при необходимости фразой — необратимость |
| **D** | Зачем | Масштаб / критерий успеха; субъектность при необходимости хвостом: «…; отвечает [роль]» |

Дублировать префикс по желанию: **Решение:** …

### MD-6. Дерево решений в проекте (если горизонт = проект и пользователь согласен писать в файл)

**Каноническая секция в Markdown:** `### Дерево решений`  
Подцели — подзаголовки четвёртого уровня: `#### Чтобы …` (краткая формулировка D).

Под каждым «Чтобы …» — маркированный список:

- **Решение 1:** полная строка A/B/C/D или краткая форма + дата/ссылка на сессию.

Если несколько независимых целей — несколько блоков `#### Чтобы D₁`, `#### Чтобы D₂`.

**Где размещать:** в карточке проекта с тегом `#project` — логично **после** блока «# Цель и стратегия» ([Templates/Project.Template.md](Templates/Project.Template.md) не содержит эту секцию по умолчанию — **создать при первом решении**). Альтернатива: раздел «Ключевые решения» в ПЗ проекта ([CLAUDE.md](CLAUDE.md)) — тогда дублировать ту же структуру или дать ссылку на карточку проекта.

Правки в файле — **только после явного согласия** пользователя (путь к файлу известен).

### MD-7. Иные фиксации

- **Стратегия:** строка A/B/C/D в документе стратегии или ПЗ; дерево по структуре документа по согласованию.
- **Задача:** строка A/B/C/D в теле Objective или в ежедневной заметке / комментарии к задаче — по запросу.
