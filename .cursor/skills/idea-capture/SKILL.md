# Навык фиксации идей (idea-capture)

**Проект:** [[186 Система достижения целей (GTD)]]. Фиксация идей сразу в **ежедневную заметку** или в **Inbox проекта** (секция `# 📥 Inbox` в файле проекта). Общий Inbox (`YYYY/00-Inbox/TrigerNN.md`) не используется.

**Связь с [project-management](.cursor/skills/project-management/SKILL.md):** при выборе «Inbox проекта» — поиск проекта по ProjectId или названию (как в обзоре проекта R-1), запись в секцию `# 📥 Inbox` файла проекта. Сценарий «Фиксация идеи» в [daily-productivity](.cursor/skills/daily-productivity/SKILL.md); команда [idea-capture](.claude/commands/idea-capture.md); субагент для контекста процесса/техники (см. ниже).

## Light-режим в 504-demo (без GTD)

В vault 504-demo нет карточек проектов GTD 186, нет общего лога идей, нет subagent-инфраструктуры. Применять **упрощённую логику**:

### Три сценария фиксации

1. **Короткая идея (1–2 строки)** → в daily-заметку `2026/01-Daily/YYYY-MM-DD.md`, секция `## Входящие дня` одной строкой: `- [ ] {текст} → Chat ⏰ HH:MM`. Без сократовского диалога, без «Зачем», без дубля в проект. Секции `## Входящие дня` может не быть — создать её после «Заметки в течение дня» или в конце файла.

2. **Развёрнутая идея (5+ строк, с заголовком)** → отдельный файл `My-Notes/{YYYY-MM-DD}-Idea-{slug}.md` + строка `- [ ] [[{YYYY-MM-DD}-Idea-{slug}|{заголовок}]]` в `## Входящие дня` daily-заметки. В отдельном файле — wikilink `Day:: [[YYYY-MM-DD]]`.

3. **Идея привязанная к модулю/навыку курса** (например, про S09 или про конкретный протокол M3) → `My-Notes/Module-{N}-Ideas.md` одной строкой или отдельный файл `My-Notes/M{N}-Idea-{slug}.md`. Дубль в daily — обязательно.

### Чего НЕ делать в 504-demo

- Не искать карточку проекта (её нет).
- Не писать в `02-CoreAutomation/186/Idea-Capture-Log.md` — этого файла нет.
- Не вызывать subagent через `mcp_task` — в Claude Code того же инструмента нет; правки в SKILL-файлах делать напрямую через Edit с подтверждением.
- Не применять полный протокол со сократовским диалогом, если пользователь не попросил углублённо.

Полный протокол ниже (режим Master-Strategy2) остаётся как fallback при работе в полном vault MS.

## Когда применять

### Триггеры (ключевые слова)

1. **«Зафиксируй идею»** (и близкие формы): «зафиксируй идею», «зафиксируй идею: …», «сохрани мысль», «запиши мысль», «quick capture идея», «зафиксируй идею по процессу X», «идея по технике Y».
2. **«Идея» в начале сообщения** — после `trim` первое слово целиком **«Идея»** (регистр как в русском, обычно с заглавной). Дальше:
   - разделитель **`:`**, **`—`**, **`–`**, **`-`** (с опциональными пробелами вокруг тире) — **текст идеи** = всё, что после разделителя, с `trim`;
   - или сразу **пробел** — **текст идеи** = всё после первого пробела после «Идея», с `trim`.
   - Если после маркера пусто (например, только «Идея:» или одно слово «Идея») — как при «зафиксируй идею» без текста: один вопрос «Напишите, что зафиксировать.»
3. Прочие явные формы: «идея: …» в любом месте сообщения (если пользователь явно метит захват), «идея по процессу/навыку X».

**Не считать захватом:** «спланируй день», «разобрать Inbox» (это daily-productivity). Фразы вроде «идея хорошая» **в середине** ответа без маркера в начале сообщения — не idea-capture. Если сомнение (одно слово после «Идея », например только оценка) — уточнить одним вопросом: «Зафиксировать как идею в дневник?»

## Куда записывать (без общего Inbox)

**Канон ежедневной заметки:**
- **Путь:** **`YYYY/01-Daily/YYYY-MM-DD.md`** (имя файла = полная дата **ISO**; год в пути = год даты). **Obsidian:** Daily notes — папка **`YYYY/01-Daily`**, имя `YYYY-MM-DD`, шаблон `Templates/Daily.Template.md` ([gtd-workflow](.cursor/rules/gtd-workflow.md)). Для «сегодня» по возможности сначала **`obsidian daily`**, затем допись.
- **Секция для захвата идей и быстрых входящих:** **`## Входящие дня`**. При отсутствии секции — добавить после блока «День» или перед `# Завершение Review` (как в шаблоне). **Legacy:** в старых файлах могла быть `### 💡 Идеи` — при записи переносить содержимое в «Входящие дня» или писать в «Входящие дня» и не плодить дубли.
- **Legacy:** папка `01 Dayly/` удалена (миграция 2026-03); все записи — в `YYYY/01-Daily/`.

1. **Ежедневная заметка** — по канону выше.
2. **Inbox проекта** — секция `# 📥 Inbox` в файле проекта. Проект определяется по ProjectId или названию (поиск как в [project-management](.cursor/skills/project-management/SKILL.md), R-1: файлы с `#project` и `ProjectId: N` в frontmatter). Если секции `# 📥 Inbox` в файле проекта нет — создать её (по [258-CRUD](02-CoreAutomation/258/258-CRUD-Правила работы с сущностями.md): у каждой сущности есть секция Inbox).

## Протокол (основной поток)

**Один шаг за ответ; ответ заканчивается вопросом или подтверждением.**

### Шаг 1. Получить текст идеи

- Если сообщение начинается с триггера **«Идея»** (см. раздел «Когда применять») — взять **текст идеи** по правилам извлечения после маркера; пустой остаток → вопрос «Напишите, что зафиксировать.»
- Если текст в сообщении («зафиксируй идею: сделать X») — использовать его (после двоеточия или целиком фразу без служебной части запроса — по смыслу).
- Если только «зафиксируй идею» / «сохрани мысль» — один вопрос: «Напишите, что зафиксировать.»
- При вызове субагента с контекстом процесса/техники: контекст передан; в записи указать источник (ссылка на процесс/технику).
- После появления текста: если подходит **шаг 1в** (большая идея) — сначала ветка 1в; иначе — шаг 1б (опционально) → шаг 2.

### Шаг 1б. Сократовский диалог (опционально)

Не начинать с пустого вопроса «зачем?» — сначала формулировать предположение, потом задавать вопрос.

- **1) Предположение по направлению и проекту:** по тексту идеи и контексту (чат, открытые файлы, упомянутые проекты) сформулировать краткое предположение: к какому направлению/сфере и к какому проекту это может относиться. Затем один вопрос на подтверждение или уточнение (например: «Похоже, это про [направление] и может относиться к проекту [X]. Верно?»). Ответ пользователя уточняет место записи (проект) или оставляет выбор на шаге 2.
- **2) Предположение по ценности реализации:** сформулировать краткое предположение: зачем это делать, какую проблему или возможность это закрывает, что даст реализация. Затем один вопрос на подтверждение или уточнение (например: «Ценность реализации, как я понимаю, — [кратко]. Что бы вы добавили или изменили?»). Ответ по желанию включается в запись как «Зачем».
- **Когда применять:** после получения текста идеи (шаг 1), перед записью (шаг 3). Опционально — если пользователь не попросил «быстро» / «просто запиши».
- **Когда не применять:** при явном запросе на скорость («быстро», «просто зафиксируй», «без лишнего»); при вызове субагента с жёстким контекстом можно опустить или только одно предположение.
- Если пользователь ответил — включить кратко «зачем» (и при необходимости проект/направление) в запись на шаге 3. Если не ответил или «пропустим» — записывать только текст идеи.

### Шаг 1в. Большая идея — отдельная заметка (ветка)

**Когда:** несколько абзацев текста; явный запрос «вынеси в заметку» / «отдельный файл»; масштаб как **Change** в логике субагента (структура, несколько разделов, отдельная проработка). **Не** применять при запросе «быстро» / «одной строкой в день».

**Поток (один шаг за ответ):**

1. Согласовать **заголовок** заметки и **тип** (черновик note / source / ПЗ — ориентир [258-Notes-Management-Policy](02-CoreAutomation/258/258-Notes-Management-Policy.md); без развёрнутого шаблона 258 при захвате).
2. **Путь нового файла:** `{YYYY}/{YYYY-MM-DD}-Idea-{slug}.md`, где `YYYY` и дата — от **целевого дня** записи, `slug` — краткое латиницей/транслит (без пробелов, без спецсимволов). Папка года должна существовать или создаётся при записи.
3. Создать заметку: минимальный YAML (`tags`, `date`/`CDate` по необходимости), заголовок `# …`, тело — текст идеи (можно сжато); внизу или в frontmatter связь с днём: строка `Day:: [[YYYY-MM-DD]]` (wikilink = **имя файла** ежедневной заметки).
4. В **`YYYY/01-Daily/YYYY-MM-DD.md`** в **`## Входящие дня`** добавить строку **`- [ ] [[Заголовок заметки]]`** (точное имя для wikilink).
5. Дописать ту же строку (или ссылку) в [Idea-Capture-Log](02-CoreAutomation/186/Idea-Capture-Log.md) по неделе `## YYYY-Wxx`.
6. Подтверждение пользователю: путь к новой заметке + путь к дню. **Шаги 2–3 основного потока** для этой идеи не выполнять (место записи уже определено).

Если размер **не** «большой» — перейти к шагу 2.

### Шаг 2. Определить место записи

- Если пользователь уже указал («в ежедневную», «в проект 277», «идея для проекта X») — использовать это.
- **Если проект не указан** (нет ProjectId / названия карточки / явного «в Inbox проекта») — **по умолчанию ежедневная заметка** на дату фиксации: `YYYY/01-Daily/YYYY-MM-DD.md` → `## Входящие дня`. Вопрос «куда записать» не задавать.
- Если пользователь **явно** просит выбрать место или формулировка отсылает к конкретному проекту без номера — один уточняющий вопрос (ежедневная vs проект и какой ProjectId/название).
- **Inbox проекта:** найти файл проекта по ProjectId или названию (Grep по `ProjectId: N` или по названию в заголовке/frontmatter), как в навыке project-management. Если проект указан, но не найден — сообщить и **записать в ежедневную заметку** на эту дату (fallback), предложив уточнить проект при следующей фиксации.

### Шаг 3. Записать идею

**Базовый формат строки в «Входящие дня»:** `- [ ] {краткий текст; опционально эмодзи 💡} → {источник} ⏰ HH:MM`. Источник: `Chat`, ссылка на процесс/технику `[[…]]` или «Ежедневная заметка» / проект.

**Если в шаге 1б получен ответ «зачем» (ценность реализации):** записать двумя строками — основная строка с чекбоксом, под ней отступом подпункт:
```markdown
- [ ] {текст идеи} → {источник} ⏰ HH:MM
  - *Зачем:* {краткий ответ пользователя}
```
Если «зачем» не формулировали — только одна строка в базовом формате.

- **В ежедневную заметку:** путь **`YYYY/01-Daily/YYYY-MM-DD.md`**. Добавить строку в **`## Входящие дня`** (если секции нет — создать заголовок). Если файла нет — `obsidian daily` (сегодня) или минимальный каркас по [daily-productivity](.cursor/skills/daily-productivity/SKILL.md) шаг 0 в **`YYYY/01-Daily/`**.
- **В Inbox проекта:** в файле проекта (найденном на шаге 2) в секцию **`# 📥 Inbox`** или эквивалентную секцию Inbox в карточке (например `## 📥 Inbox …`) дописать строку(и). Если секции нет — добавить `# 📥 Inbox` и под ней строку. Опционально в конце основной строки: `[projectId:: NNN]`.
- **Дубль в день при записи в Inbox проекта (обязательно):** в тот же календарный день фиксации добавить в **`YYYY/01-Daily/YYYY-MM-DD.md`** → **`## Входящие дня`** **короткую** строку в базовом формате с тем же смыслом, а в поле источника указать ссылку на карточку проекта (`→ [[…]]`), чтобы ежедневный debrief и [daily-productivity](.cursor/skills/daily-productivity/SKILL.md) видели захват без открытия проекта. Полная формулировка остаётся в Inbox проекта; в дне допустима сжатая версия.
- **Логирование для анализа:** дополнительно дописать ту же запись (те же строка/строки) в единый лог идей — файл [02-CoreAutomation/186/Idea-Capture-Log.md](02-CoreAutomation/186/Idea-Capture-Log.md). В логе — секция по неделе `## YYYY-Wxx` (ISO week); при отсутствии секции для текущей недели создать заголовок и добавить запись. Лог используется для еженедельного обзора и улучшения работы навыка (см. регулярное действие в Guide - Недельные обзоры и задачу 186-19).

### Шаг 4. Подтверждение

- Кратко: «Записал в {путь к файлу} и в лог идей (Idea-Capture-Log).» Если была запись **в Inbox проекта** — явно указать также **дубль в ежедневную заметку** `{YYYY/01-Daily/YYYY-MM-DD.md}`. Завершение: «При желании можно превратить в задачу или Backlog (через project-management или gtd-workflow).»

### Шаг 5. Опционально — сразу в Backlog

- Если пользователь просил «сразу в Backlog» / «и в Backlog»: по [gtd-workflow](.cursor/rules/gtd-workflow.md), раздел «Обработка новых идей» — определить Sphere, найти открытый проект или Backlog домена, добавить пункт (краткое описание + гипотеза + `[created:: YYYY-MM-DD]`). Идею в ежедневную заметку или Inbox проекта всё равно зафиксировать (шаг 3).

## Вызов как субагент (процесс / техника)

Когда пользователь указывает процесс или технику («зафиксируй идею по процессу X», «идея по технике Y») или идея фиксируется в контексте открытого документа процесса/техники:

1. **Инициатор** передаёт субагенту контекст: путь к файлу процесса/техники или название.
2. **Субагент** выполняет протокол idea-capture; в записи указывает источник: `→ [[Имя процесса или техники]]`. Место записи — по умолчанию ежедневная заметка, либо пользователь указывает проект.
3. Текст идеи субагент запрашивает у пользователя или получает из сообщения инициатора.

**Формулировка задачи для mcp_task (generalPurpose):** «Пользователь хочет зафиксировать идею в контексте процесса/техники. Контекст: [путь или название]. Выполни протокол idea-capture: получи текст идеи, сохрани в `YYYY/01-Daily/YYYY-MM-DD.md` → секция `## Входящие дня` или Inbox проекта, в записи укажи источник → [[имя процесса/техники]]. Верни подтверждение и путь к файлу.»

## Субагент: идея улучшения процесса/навыка во время работы (Cursor / CC)

**Правило: действие Fix (внести правку в файл процесса/навыка) всегда выполняется в субагенте.** Основной агент не вносит правку сам — запускает субагент (mcp_task, generalPurpose) с задачей: оценить масштаб и либо зафиксировать идею, либо внести правку в целевой файл. Анализ и действие выполняются **внутри субагента**.

**Контекст:** при работе над задачей в Cursor или Claude Code (CC) возникла идея улучшения процесса или навыка (например, правка в SKILL.md, Guide, описании процесса). Пользователь может сразу запустить субагент; основной агент при запросе на правку/улучшение всегда делегирует выполнение субагенту.

**Логика субагента:**

1. **Получить от инициатора:** контекст (путь к файлу процесса/навыка или название), формулировку идеи улучшения (или запросить у пользователя).
2. **Проанализировать масштаб изменения:**
   - **Большое изменение (Change):** затрагивает несколько файлов или разделов; меняет структуру протокола, сценарии, шаги; добавляет новый сценарий или значимый блок; требует обсуждения или отдельной задачи. → **Фиксировать идею** по протоколу idea-capture (ежедневная заметка или Inbox проекта; источник — ссылка на процесс/навык). Не вносить правку самому.
   - **Маленькое изменение:** одна формулировка, один пункт списка, исправление опечатки, уточнение одной фразы, добавление одного блока в один файл — можно внести без риска для целостности. → **Внести правку** в целевой файл (процесс, навык, Guide); при необходимости кратко подтвердить пользователю.
3. **Выполнить:** либо запись идеи (шаги 2–4 основного протокола idea-capture), либо точечное редактирование файла. Вернуть инициатору краткий отчёт: что сделано (идея зафиксирована в {путь} / правка внесена в {файл}).

**Когда запускать:** пользователь говорит «улучшить процесс», «идея по навыку», «зафиксируй улучшение» или «внеси правку в процесс X» в контексте работы над задачей; или основной агент предлагает: «Запустить субагент: проанализирует масштаб и либо зафиксирует идею, либо внесёт правку?»

**Формулировка задачи для mcp_task (generalPurpose):** «При работе над задачей возникла идея улучшения процесса/навыка. Контекст: [путь к файлу или название процесса/навыка]. Идея: [формулировка]. Выполни в субагенте: 1) Оцени масштаб: большое изменение (Change) — несколько файлов/разделов, новая структура, новый сценарий → зафиксировать по idea-capture: «Входящие дня» в `YYYY/01-Daily/YYYY-MM-DD.md` или отдельная заметка (шаг 1в), или Inbox проекта; источник = этот процесс/навык. Маленькое изменение — одна формулировка, один пункт, опечатка, одна фраза → внести правку в целевой файл. 2) Выполни выбранное действие. 3) Верни отчёт: что сделано и куда.»

---

## Связь с project-management

- **Поиск проекта:** при выборе «Inbox проекта» использовать тот же способ, что в [project-management](.cursor/skills/project-management/SKILL.md) (R-1): поиск по воркспейсу файлов с тегом `#project` и frontmatter `ProjectId: N` (или по названию проекта). Один проект — запись в его файл; несколько кандидатов — уточнить у пользователя.
- **Дальнейшие действия:** превращение идеи в Objective/Task или Backlog — через навык project-management или по gtd-workflow (раздел «Обработка новых идей»). Навык idea-capture только фиксирует идею в выбранное место.

---

## Что не делать

- Не записывать в общий Inbox (`YYYY/00-Inbox/TrigerNN.md`).
- Не разворачивать при захвате полный шаблон 258 (гипотеза, плюсы/минусы, MoSCoW, timing).
- **Не вносить правку (Fix) в файл процесса/навыка самому** — действие Fix всегда выполнять в субагенте (запуск mcp_task с задачей оценить масштаб и внести правку или зафиксировать идею).
- Не задавать вопрос «куда записать», если проект не указан — тогда сразу ежедневная заметка (см. шаг 2). Уточняющий вопрос по месту — только при явной просьбе выбрать или неоднозначности. Сократовские вопросы (шаг 1б) — опционально 1–2, не превращать в длинный диалог.
