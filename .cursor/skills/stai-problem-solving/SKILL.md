
## Входные данные

Пользователь предоставляет:

- **Нежелательные явления** — что беспокоит
- **Стейкхолдеры** — кто затронут

Если данных не хватает — запроси.

## Фаза 1: Формулирование проблемы

### Шаги

1. **Проблематизация** — заполни таблицу 5 элементов:

| # | Элемент | Вопрос |
|---|---------|--------|
| 1 | Нежелательные явления | Что беспокоит? |
| 2 | Почему важно | Какой ущерб? |
| 3 | Контекст | Условия, ограничения |
| 4 | Стейкхолдеры | Кто вовлечён? |
| 5 | Желаемая картина | Как выглядит «решено»? |

2. **Формулировка gap**: As-is → To-be → Gap
3. **Классификация**: Собиратель / Охотник / Координатор (× Cynefin); *устар. синоним: «Строитель»*
4. **2-3 цепочки «5 Почему»** → определить тип корневой причины: простая / убеждения / динамика

### Д1 ⛔ Стоп-точка

Покажи: таблицу 5 элементов, gap, тип задачи, цепочки «5 почему», тип корневой причины.
Спроси: согласен ли? нужны ли правки?

> Если тип задачи = Собиратель → предложи решение напрямую, без фаз 2-3.

---

## Фаза 2: Диагностика

### Маршрутизация

```
Тип корневой причины (из Фазы 1)
├── Простая → 5 Why углублённый
├── Убеждения → Iceberg Model (4 уровня × стейкхолдеры)
├── Динамика → CLD (переменные, R/B-петли, delays)
└── Смешанный → Iceberg → CLD
```

### Шаги

1. Построить системную модель (Iceberg и/или CLD)
2. Для **Iceberg:** после согласования текстовой структуры — вывести **SVG-таблицу**
3. Распознать архетип: Limits to Growth / Shifting the Burden / Fixes that Fail / Eroding Goals / Success to Successful
4. Определить типичную точку рычага архетипа

**CLD-нотация:** `A →(+) B` (однонаправленная), `A →(−) B` (обратная), `→||(+)` (delay). R = чётное число (−), B = нечётное.

### Д2 ⛔ Стоп-точка

Покажи: модель (Iceberg/CLD), **для Iceberg — SVG-таблицу** после согласования текстовой структуры, архетип, точку рычага.
Спроси: узнаёт ли пользователь свою систему? что упущено?

---

## Фаза 3: Решение

### Шаги

1. **5 уровней рычага Медоуз** для каждой точки воздействия:
   - 1: Параметры (слабый, простой)
   - 2: Буферы/Stocks (слабый+)
   - 3: Структура — правила, информационные потоки (средний)
   - 4: Цели системы (сильный, сложный)
   - 5: Парадигмы — ментальные модели (очень сильный, очень сложный)

2. **Карта стейкхолдеров** — матрица влияние × интерес (4 квадранта)
3. **Приоритизация** — матрица Effort / Impact
4. **Портфель 3-5 гипотез** — таблица:

| # | Гипотеза | Действие | Уровень рычага | Effort | Impact |
|---|----------|----------|---------------|--------|--------|

Правила: Quick Wins (1-2) + стратегические (3-5). Минимум 1 гипотеза уровня 3+.

### Д3 ⛔ Стоп-точка

Покажи: портфель, карту стейкхолдеров, приоритизацию.
Спроси: реалистично? какие приоритеты? нужна Фаза 4?

---

## Фаза 4: Реализация (опционально)

### Шаги

1. **Дизайн экспериментов**: Plan → Test → Metric → Decision
2. **Грозовая туча** (при конфликте): общая цель → 2 потребности → условия → скрытое допущение
3. **Риск-анализ**: 3 риска × вероятность × последствия × митигация
4. **Anti-hallucination check**: данные vs мнения, confirmation bias, альтернативы

**Стратегическое расширение** (по запросу):
- Creative Tension (видение 3г / реальность / разрыв / ставки)
- Сценарное мышление (2×2 матрица неопределённостей)
- Стратегические ставки (обоснованность, необратимость, масштаб, субъектность)

### Д4 ⛔ Стоп-точка

Покажи: планы экспериментов, риски, (стратегическое расширение).
Спроси: готов к запуску?

## Режим коротких итераций

По умолчанию pipeline работает **короткими итерациями**: на каждом шаге внутри фазы показывай промежуточный результат и спрашивай подтверждение.

- После каждого шага — покажи результат, спроси: «Верно? Корректировки?»
- Если пользователь даёт уточнение — интегрируй и покажи обновлённую версию
- Если пользователь говорит «дальше» / «ок» — переходи к следующему шагу
- Допустимо объединять 2 логически связанных шага

## Два трека решений

Когда ситуация требует и коррекции, и системного улучшения:

| Трек | Горизонт | Фокус |
|------|----------|-------|
| **A: Коррекция сейчас** | Оставшиеся занятия текущего запуска | Quick wins, работа с тем что есть |
| **B: Редизайн на будущее** | Следующий запуск курса | Структурные изменения, новые механики |

Для каждого трека — отдельный портфель гипотез.

## Пояснительная записка сессии

При запуске pipeline создай `{Тема}-ПЗ-Problem-Solving.md` и обновляй после каждой фазы.

**Frontmatter:** tags: [analysis, problem-solving], gd_sync: true, ProjectId, P_date.

**Структура каждой фазы в ПЗ:**
- **Результат** — полный вывод фазы (все таблицы, диаграммы, портфели — без сокращений). ПЗ = самодостаточный документ, читатель не должен обращаться к чату
- **Анализ ответов пользователя** — что сказал, почему это важно, какой инсайт за корректировкой, как изменило модель/направление анализа

**Общая структура:** Контекст → Ключевые решения → Ход работы (4 фазы × Результат + Анализ ответов) → Открытые вопросы → Артефакты.

**Правила:**
- Старт: создай ПЗ с Контекстом
- После каждой Д-точки: заполни «Результат» полным выводом + «Анализ ответов» с объяснением логики корректировок
- Завершение: заполни «Открытые вопросы» и «Артефакты»
- Принцип полноты: не «см. выше» — дублировать ключевые артефакты целиком

## Принципы

- Следуй алгоритмам из канонов навыков точно
- На каждой Д-точке останавливайся и жди подтверждения
- Работай короткими итерациями — показывай промежуточные результаты
- Парадокс рычага: сильнее рычаг → сложнее использовать
- Предупреждение Медоуз: правильная точка, неправильное направление
- «Нет субъекта — нет проблемы» (Федин)
- Не подменяй проблему решением
