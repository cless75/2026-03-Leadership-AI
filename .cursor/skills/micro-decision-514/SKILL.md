# Навык micro-decision-514 (ОНМС + формула A/B/C/D)

**Справочник (примеры формулы):** [reference.md](reference.md).
**Команда:** [.claude/commands/micro-decision-514.md](.claude/commands/micro-decision-514.md).

## Назначение

Быстро структурировать выбор между вариантами и зафиксировать решение в виде одной читаемой строки с обоснованием. Опорный канон — **этот SKILL.md**; таблица ОНМС и формула A/B/C/D заданы ниже и достаточны для применения.

## Когда применять

По **явному запросу** или когда участник застрял между 2+ вариантами и нужна структура выбора без полного STAI pipeline.

**Триггеры (RU):** принять решение, зафиксировать решение, выбрать вариант, оценить ставку, дерево решений, ОНМС, микро-решение 514.

**Триггеры (EN):** decide, choose between, fix the decision, decision tree, bet evaluation.

**Не подменять этим навыком:**

- Полный цикл «проблема → диагностика → портфель» — **stai-problem-solving**.
- Фиксация рабочей сессии — **session-link**.
- Обновление сквозного профиля — **ms-participant-profile**.

## Куда записывается решение

### Сценарий 1 — решение дня (по умолчанию)

Блок в daily-заметке `YYYY/01-Daily/YYYY-MM-DD.md`:

```markdown
### Решение HH:mm — <короткая тема>

**Выбираем A, а не B, потому что C, чтобы D.**

| Поле | Значение |
|---|---|
| Обоснованность | … |
| Необратимость | … |
| Масштаб | … |
| Субъектность | … |
```

### Сценарий 2 — журнал решений (для накопления паттернов)

Отдельный файл `My-Notes/Diagnostics/decisions-log.md` (или `My-Notes/Logs/decisions-log.md` по выбору участника) с датированными секциями.

**Frontmatter файла журнала** (создаётся при первом решении, потом не меняется):

```yaml
---
privacy: personal
kind: diagnostic
contains-pii: no
commercial-secrets: none
mentor-shareable: yes-after-mask
tags: [decisions, 504]
---
```

Auto-suggest: если решение касается людей (увольнение, повышение, переговоры) → `commercial-secrets: possible`. Если это решение exit/entry в компанию → `mentor-shareable: yes-after-mask`.

Структура секции в журнале:

```markdown
## 2026-04-25 HH:mm — <короткая тема>

**Выбираем A, а не B, потому что C, чтобы D.**

| Поле | Значение |
|---|---|
| Обоснованность | … |
| Необратимость | … |
| Масштаб | … |
| Субъектность | … |
```

Полезно, когда участник хочет периодически перечитывать свои решения и замечать паттерны (что избегаю, куда иду, где ошибался).

### Сценарий 3 — решение внутри HW-Protocol

Если решение непосредственно связано с домашним заданием модуля — вписать блок (как Сценарий 1) в `My-Notes/M{N}-HW-Protocol.md` в секцию «Решения» или рядом с обсуждением, к которому логически относится.

### Выбор сценария

- По умолчанию — **Сценарий 1** (в daily).
- При запросе «в журнал решений» / «в decisions-log» — **Сценарий 2**.
- Если контекст — HW конкретного модуля и участник уже открыл его протокол — **Сценарий 3**.

## Режим диалога

- По умолчанию: **один логический шаг за ответ**, в конце — один уточняющий вопрос.
- Если участник просит **экспресс** на уровне задачи — можно выдать полный проход (варианты → ОНМС → формула) в одном сообщении.

## Алгоритм

### MD-0. Горизонт

Уточнить одним вопросом (если не ясно из контекста):

- **Стратегия** — ставка уровня направления / карьеры / большого проекта.
- **Проект / модуль курса** — выбор в рамках модуля или конкретной рабочей темы.
- **Задача** — тактический шаг.

Горизонт влияет на глубину заполнения ОНМС (для «задачи» — кратко, для «стратегии» — развёрнуто).

### MD-1. Вопрос решения

Короткая формулировка: *что именно решаем одним выбором?*

### MD-2. Варианты

**Минимум два**, включая явное «не делать» / «отложить», если уместно.

### MD-3. Опционально — Effort / Impact / рычаг

Если вариантов несколько и нужна приоритизация:

- **Уровень рычага** (1–5 по Медоуз) — если применимо, иначе `n/a`.
- **Effort / Impact** — низкий / средний / высокий.

### MD-4. ОНМС по выбранному варианту

Заполнить таблицу:

| Поле | Вопрос |
|---|---|
| **Обоснованность** | На чём основана ставка? Какие данные, наблюдения, модели поддерживают? |
| **Необратимость** | Насколько легко откатить? (высокая необратимость = больше проверки до запуска) |
| **Масштаб** | Какой потенциальный эффект? Масштабировать под горизонт: для задачи — на результат недели; для проекта — на цель проекта; для стратегии — на жизнь / карьеру / направление |
| **Субъектность** | Кто реализует? Личный / командный уровень? Отвечает за результат — кто? |

### MD-5. Формула фиксации A / B / C / D (обязательно)

После согласования — вывести **одну строку**:

> **Выбираем A, а не B, потому что C, чтобы D.**

| Слот | Смысл | Связь с ОНМС |
|---|---|---|
| **A** | Принятый вариант | Что делаем |
| **B** | Отвергнутый (или «не делать X») | Явный trade-off |
| **C** | Почему так | Прежде всего обоснованность; при необходимости — необратимость |
| **D** | Зачем | Масштаб / критерий успеха; субъектность хвостом при необходимости |

Примеры формулы — в [reference.md](reference.md).

### MD-6. Запись

Предложить один из трёх сценариев (см. выше), показать черновик блока, записать после «да».

Правки в vault — **только после явного согласия** (путь к файлу показан).
