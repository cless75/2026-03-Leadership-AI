# Навык session-link (старт / промежуток / финал рабочей сессии)

**Справочник:** [reference.md](reference.md).
**Команды:** [.claude/commands/session-start.md](.claude/commands/session-start.md), [.claude/commands/session-close.md](.claude/commands/session-close.md), [.claude/commands/session-result.md](.claude/commands/session-result.md).

## Назначение

Коротко фиксировать работу участника в **ежедневной заметке** так, чтобы:
- одним взглядом видеть, над какими темами шла работа, в каком состоянии, когда был последний заход;
- при обрыве сессии (смена чата, краш, исчерпание контекста) можно было продолжить с того же места через **чат-мост** `.claude/session-handoff.md`.

Навык ориентирован на лёгкий формат пакета участника: **нет** карточек Objective, **нет** ПЗ проекта, **нет** отдельного файла `## Ключевой файл сессии (ctx)`. Единственное хранилище сессии — daily-note + опциональный handoff-файл.

## Ключевые файлы

| Файл | Роль |
|---|---|
| `YYYY/01-Daily/YYYY-MM-DD.md` | Журнал сессий в секции `## Заметки в течение дня` |
| `.claude/session-handoff.md` | Чат-мост: компактный «где мы остановились» для следующей AI-сессии |

---

## Тема сессии (вместо Objective)

В лёгком пакете роль Objective играет **тема** — одна фраза «над чем работаем». Пример: `Подготовка HW M3`, `Разбор конфликта с PO`, `S09 value chain для своей команды`.

- Тема — произвольный текст; используется как wikilink в заголовке блока дня: `[[Подготовка HW M3]]`.
- Если участник хочет постоянный файл по теме — создаёт `My-Notes/<тема>.md` сам (навык этого не требует, но поддерживает ссылку, если файл есть).
- Тема не обязана существовать как файл — wikilink работает и на несуществующий файл (Obsidian «пустая» ссылка).

## Статусы в заголовке блока (эмодзи в скобках)

| Эмодзи | Статус | Когда ставить |
|---|---|---|
| 🔵 | Старт | Первый заход по этой теме в этот календарный день |
| 🟡 | Продолжение | Повторный заход по той же теме в тот же день (новый `###` с новым `HH:mm`) |
| 🟢 | Выполнено | Работа по теме завершена |
| 🟣 | DoD на сегодня завершён | На сегодня достигли цели; тема может быть открыта на другие дни |
| ⚪ | Отложена | Осознанная пауза |
| 🔴 | Не сработало | Попытка не дала результата |

---

## Когда применять

| Формулировка пользователя | Режим |
|---|---|
| «работаем над», «начинаем сессию», «session start», «только старт», «отметь открытие» | START |
| «промежуточный итог», «checkpoint», «сохрани состояние», «пока не закрываем» | MID |
| «зафиксируй результат», «итог», «закрываем», «session close», «session result», «handoff» | CLOSE |

Если пользователь пишет только «зафиксируй сессию» без уточнения — **один вопрос**: «Фиксируем старт, промежуточный итог или финал?». Не угадывать молча.

Если в файле дня уже есть открытый блок 🔵/🟡 по той же теме и пользователь приносит итог — по умолчанию **CLOSE** для этого блока (не плодить второй блок).

## Подпротокол START (4 шага)

### START-1. Дата

Сегодня / `YYYY-MM-DD`. Файл дня: `YYYY/01-Daily/YYYY-MM-DD.md`. Если файла нет — создать минимальный:

```markdown
# YYYY-MM-DD

## Заметки в течение дня
```

### START-2. Тема

Одна фраза «над чем работаем». Если не указана — **один** вопрос: «Над какой темой стартуем?». Допустимы ответы «пока не знаю / разберёмся по ходу» — тогда тема = кратко из контекста чата, с пометкой «уточнить».

### START-3. Черновик блока

В черновике для подтверждения показать:

```markdown
### HH:mm [[<тема>]] (🔵 Старт)
- [ ] Начало работы · Claude Code
```

Если по этой теме уже есть блок 🔵 или 🟡 в этом же дне — показать черновик как 🟡 Продолжение.

### START-4. Запись после «да»

После подтверждения пользователя — дописать блок в конец секции `## Заметки в течение дня`. Если секции нет — создать.

Короткое подтверждение: «Записал старт сессии в [[YYYY-MM-DD]]».

---

## Подпротокол MID (3 шага)

### MID-1. Найти открытый блок

В `## Заметки в течение дня` найти **последний** блок `### HH:mm [[<тема>]] (🔵 Старт)` или `(🟡 Продолжение)` по соответствующей теме.

Если открытого блока нет — спросить: «Открытого блока по этой теме в сегодняшнем дне нет. Ставим продолжение (новый 🟡-блок) или это новая тема (🔵 Старт)?».

### MID-2. Черновик двух строк

Под найденным заголовком предложить дописать:

```markdown
- [x] <кратко что сделано>
- [ ] Следующий шаг: <кратко>
```

Строку «Следующий шаг» опустить, если пользователь явно не хочет фиксировать.

### MID-3. Запись после «да»

Дописать строки под найденным заголовком. **Статус в скобках не меняем** (🔵 или 🟡 остаётся).

Короткое подтверждение.

---

## Подпротокол CLOSE (5 шагов)

### CLOSE-1. Найти открытый блок

Как MID-1. Если блока нет и пользователь приносит итог — сначала START (с пониженным временем, соответствующим фактическому), затем сразу CLOSE.

### CLOSE-2. Черновик итога

Под заголовком предложить:

```markdown
- [x] Итог: <одна фраза>
```

Опционально блок:

```markdown
**Следующие шаги (YYYY-MM-DD):**
- …
- …
```

### CLOSE-3. Обязательный вопрос о следующих шагах

> «Какие 1–3 следующих шага записать? Если нет — ответьте «нет» или «пропуск».»

Ждать ответ. Если «нет» / «пропуск» — блок «Следующие шаги» не добавлять.

### CLOSE-4. Смена статуса в заголовке

Заменить скобки `(🔵 Старт)` / `(🟡 Продолжение)` на финальный:
- `(🟢 Выполнено)` — работа по теме завершена;
- `(🟣 DoD на сегодня завершён)` — на сегодня цель достигнута, тема может продолжиться;
- `(⚪ Отложена)` — осознанная пауза;
- `(🔴 Не сработало)` — попытка без результата.

### CLOSE-5. Чат-мост (по умолчанию)

Дописать блок в `.claude/session-handoff.md`:

```markdown
## Session handoff YYYY-MM-DD HH:mm

- Тема: <тема>
- Сделано:
  - …
  - …
- Следующие шаги:
  - …
- Связанные файлы: [[…]], [[…]]
```

**Opt-out:** если пользователь сказал «только daily» / «без handoff» — пропустить.

Короткое подтверждение: «Сессия закрыта. Статус 🟢. Handoff дописан.»

## Один шаг за ответ

Если требуется выбор пользователя (дата, тема, следующие шаги, статус) — **один вопрос за ответ**. Правки в daily и handoff — только после явного «да» / «запиши».

---

## Mid-Course Diagnostic Gate (для курсов с M3+)

Перед выполнением **START** по модулю **M3** или позже в рамках курса 504 (и аналогичных self-paced программ MS) — проверить:

1. Определён ли `MS_PROFILE_PATH` в `.claude/settings.local.json`? Если нет — предложить `/profile-intake` до старта сессии.
2. Существует ли в `MS_PROFILE_PATH` или рядом файл `mid-course-diagnostic-*.md` за последние 30 дней? Если **нет** и участник уже в M3+ — **заблокировать** START одним сообщением:
   > «Ты начинаешь работу в M3+, но Mid-Course Diagnostic (обязательный checkpoint после M2) либо не пройден, либо старше 30 дней. Он защищает от случая, когда цели программы сместились, пока ты внутри. Запустить `/mid-course-diagnostic` сейчас? (45–60 мин).
   >
   > Пропустить — только явно: напиши «пропустить checkpoint» (запись попадёт в `ms-profile.md` → раздел «Пропущенные checkpoints»).»
3. При явном `пропустить checkpoint` — дописать в `ms-profile.md` секцию:
   ```
   ## Пропущенные checkpoints
   - Программа: 504, Module: M3 start, Причина: <участник указал>, Дата: YYYY-MM-DD
   ```
   И продолжить со START.
4. При согласии пользователя — передать управление в `/mid-course-diagnostic` (навык `ms-participant-profile`, режим 3). После завершения diagnostic — вернуться к исходному запросу START автоматически.

**Gate не срабатывает** если:
- участник явно в M1/M2 (checkpoint ещё не время);
- в запросе нет упоминания M3+ и не открыт файл `Module-3/` или выше;
- Diagnostic пройден за последние 30 дней.

## Burnout-сигналы в START/MID/CLOSE

Агент **проверяет сигналы выгорания** в текстах сессии и daily-notes. Конкретные маркеры — в [burnout-self-diagnosis/SKILL.md](../burnout-self-diagnosis/SKILL.md), раздел «Автотриггер в других навыках». Триггерные сигналы в session-link:

- **В START:** тема сессии содержит «не могу», «всё на автомате», «тело голосует», «ничего не хочется»; или участник стартует новую большую тему при уже открытых 3+ незакрытых блоках 🔵/🟡 прошлых дней.
- **В MID:** 3+ сессии подряд с `- [x]` «сделано, но через силу», «еле», «на зубах»; промежуточные итоги через пустое содержимое.
- **В CLOSE:** 3+ сессий подряд закрыты статусом 🔴 «Не сработало» за 7 дней; или ⚪ «Отложена» с формулировкой «нет ресурса», «устал(а)», «не вытягиваю».

При триггере — **одно мягкое сообщение** в рамках текущего шага:

> Я заметил сигналы, которые описаны в burnout-self-diagnosis как маркеры истощения: <цитаты>. Не срочно сейчас, но если ты не проходил `/burnout-check` в последние 2–4 недели — стоит сделать 5-минутный срез. Это не отменяет текущую сессию, просто пометка.

Не блокировать текущую START/MID/CLOSE. Но если результат предыдущего `/burnout-check` = 🟠 или 🔴 и участник начинает **новую большую инициативу** (exit, новый проект, смена курса) — **мягкий блок** по аналогии с Mid-Course Gate с предложением сначала повторного re-check или разговора с ментором.

## Что не делать

- **Не** искать карточку Objective (её нет в light-пакете).
- **Не** создавать/править ПЗ (её нет в light-пакете).
- **Не** писать только в `session-handoff.md` без записи в daily — canonический журнал это `## Заметки в течение дня`.
- **Не** создавать секцию `## Ключевой файл сессии (ctx)` — она относится к другому (полному) формату, в light не используется.
- **Не** добавлять ссылки на `ПЗ-*.md`, `*-п-*.md`, `project-management`, `refactoring-objective` — этих сущностей в light нет.
